# Sub-Component: Encryption Utility

## 1. The Goal

Securely store Alpaca API credentials (and any future broker keys) in the database by encrypting them with AES-256-GCM before storage, ensuring plaintext keys never appear in logs, API responses, or browser bundles.

---

## 2. The Problem It Solves

API keys stored in plaintext in `user_profiles` columns would be exposed in any database breach, leaked Supabase query, or accidental log line. Encrypting them with a per-value unique IV means each ciphertext is independently secure — compromising one encrypted value does not expose others.

---

## 3. The Proposed Solution / Underlying Concept

### Algorithm: AES-256-GCM

```
ENCRYPTION_KEY: 64-character hex string (32 bytes) in Vercel environment variable
IV: 12 bytes (96 bits), randomly generated per encryption call
Auth Tag: 16 bytes (128 bits), generated by the cipher
Output format: "iv_b64:authTag_b64:data_b64" (colon-separated base64)
```

### `encrypt(plaintext, keyHex)` → string

```typescript
const iv = randomBytes(12)
const cipher = createCipheriv('aes-256-gcm', key, iv)
const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
const authTag = cipher.getAuthTag()
return `${iv.toString('base64')}:${authTag.toString('base64')}:${encrypted.toString('base64')}`
```

### `decrypt(ciphertext, keyHex)` → string

```typescript
const [ivB64, authTagB64, dataB64] = ciphertext.split(':')
const decipher = createDecipheriv('aes-256-gcm', key, Buffer.from(ivB64, 'base64'))
decipher.setAuthTag(Buffer.from(authTagB64, 'base64'))
return Buffer.concat([decipher.update(data), decipher.final()]).toString('utf8')
```

### Key Validation

```typescript
function keyBuffer(keyHex: string): Buffer {
  if (keyHex.length !== 64) {
    throw new Error('ENCRYPTION_KEY must be a 64-character hex string (32 bytes)')
  }
  return Buffer.from(keyHex, 'hex')
}
```

### Security Rules (CLAUDE.md Rule 4)

- `ENCRYPTION_KEY` stored only in Vercel environment variables — never committed to git
- Decryption happens server-side only, immediately before the API call
- Plaintext keys are never logged, returned in responses, or stored in browser state
- Each value has a unique IV — same plaintext produces different ciphertext each time

### Usage in Execute Route

```typescript
const alpacaKey = decrypt(profile.alpaca_key_enc, process.env.ENCRYPTION_KEY!)
const alpacaSecret = decrypt(profile.alpaca_secret_enc, process.env.ENCRYPTION_KEY!)
```

## 4. Testing & Verification

| What to Verify | How |
|---|---|
| Round-trip: encrypt → decrypt | `encrypt(plaintext, key)` → `decrypt(result, key)` === original plaintext |
| Wrong key throws | `decrypt(ciphertext, wrongKey)` → throws |
| Tampered ciphertext throws | Modify one character in ciphertext → GCM authentication fails |
| IV uniqueness | Encrypt same value twice → different ciphertexts |
| Key format enforced | Pass 32-byte hex string → works; pass 64-char hex → throws |
| `pnpm test` | `lib/encryption.test.ts` passes |
